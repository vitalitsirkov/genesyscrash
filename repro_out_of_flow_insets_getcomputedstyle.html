Repro: CHECK(!layout_results.empty()) — OOF getComputedStyle crash

Crash: layout_box.cc:4389 — CHECK(!layout_results.empty()) inside LayoutBox::OutOfFlowInsetsForGetComputedStyle().
Fix: computed_style_utils.cc:811 — guard !box->GetLayoutResults().empty() added before calling OutOfFlowInsetsForGetComputedStyle().

Why this page often never crashes (even on an unpatched build)
The failing CHECK needs a LayoutBox that is already out-of-flow but whose GetLayoutResults() list is still empty when inset values are resolved. In normal browsing, getComputedStyle goes through Document::UpdateStyleAndLayoutTree / UpdateStyleAndLayout; if those bail out early (e.g. render throttling at document.cc:2302), Blink usually never attaches a fresh layout box for new DOM — so ValueForPositionOffset sees layout_object == nullptr and never calls OutOfFlowInsetsForGetComputedStyle. WhatsApp’s crash is therefore probably tied to a narrower embedding or call-site that is not recreated by static HTML alone. Next step: capture a symbolized stack from the unpatched CEF build when opening chat settings and confirm the top frames match this path (or paste them for a targeted repro).

Crash anatomy

CRASH SITE (layout_box.cc:4385):
  const BoxStrut& LayoutBox::OutOfFlowInsetsForGetComputedStyle() const {
    const auto& layout_results = GetLayoutResults();
    CHECK(!layout_results.empty());   // ← fires when the LayoutBox was never laid out
    return layout_results.front()->OutOfFlowInsetsForGetComputedStyle();
  }

REQUIRED CONDITIONS:
  1. box->IsOutOfFlowPositioned() == true   (position:absolute / position:fixed / OOF-in-multicol)
  2. offset.IsFixed() == false              (top/left/right/bottom is: auto, %, calc, not a plain px value)
  3. box->GetLayoutResults().empty() == true (the LayoutBox was created but never given a layout result)

PATCHED GUARD (computed_style_utils.cc:811):
  // Before:  if (box && box->IsOutOfFlowPositioned())
  // After:   if (box && box->IsOutOfFlowPositioned() && !box->GetLayoutResults().empty())

HOW layout_results_ CAN BE EMPTY (for this CHECK):
  a) UpdateStyleForbiddenScope — Document::UpdateStyleAndLayout(/)Tree return early
     (CEF: layout_tree_scheduler, blink_helper, vrm_helper, vrm_composite_tick patch).
  b) Very specific layout-engine races (subtree / fragmentation / etc.) — need a
     stack trace to match; not covered reliably by static pages tested here.
  c) ShouldThrottleRendering() skips work at document.cc:2302 / local_frame_view
     ~3088 — usually avoids this CHECK because the LayoutBox is often not attached
     yet (layout_object null), not “OOF with empty layout_results_”.
