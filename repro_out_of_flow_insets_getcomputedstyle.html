Repro: OOF getComputedStyle crash

This crash requires the CEF-specific UpdateStyleForbiddenScope to be active. It cannot be reproduced by opening this page in a plain browser. Load this page inside the CEF 6367 build (without the fix) while the composition tick is running — i.e. after the page is registered with the layout tree but before the first paint has committed.

Root cause

CEF's LayoutTreeScheduler::OnCompositionStarted() (layout_tree_scheduler.cc:187) enters UpdateStyleForbiddenScope, which patches Document::UpdateStyleAndLayout() to return early: 

// document.cc (patched by CEF)
void Document::UpdateStyleAndLayout(DocumentUpdateReason reason) {
  ...
  if (UpdateStyleForbiddenScope::IsUpdateStyleForbidden())
    return;          // ← layout never runs inside a composition tick
  ...

When page JS calls getComputedStyle(el).top on an out-of-flow–positioned element while a composition tick is in progress (e.g. from a MutationObserver, a layout-tree event callback, or simply an element inserted just before a tick fires), the call chain is: 

getComputedStyle(el).top
  → CSSComputedStyleDeclaration::GetPropertyCSSValue("top")
    → UpdateStyleAndLayoutIfNeeded()         // IsLayoutDependent → true
      → Document::UpdateStyleAndLayoutForNode()
        → Document::UpdateStyleAndLayout()
          → returns early (UpdateStyleForbiddenScope)
    → Bottom/Top::CSSValueFromComputedStyleInternal()
      → ComputedStyleUtils::ValueForPositionOffset()
        // offset is 'auto' or '%' → IsFixed() is false → no early-return at line 807
        if (box && box->IsOutOfFlowPositioned())  // ← TRUE (element is abspos)
          box->OutOfFlowInsetsForGetComputedStyle()
            CHECK(!layout_results.empty())  // ← CRASH: box never laid out

Trigger conditions (both required)

  1. UpdateStyleForbiddenScope is held — happens inside CEF's composition tick (LayoutTreeScheduler::OnCompositionStarted), VRMHelper::GetElementLeftAndTop, BlinkHelper::GetElementPositionFromViewPortById, and several other CEF helpers.
  2. OOF element has non-fixed inset — top: auto, top: 50%, top: calc(...), etc. A fixed value like top: 10px takes an early-return at line 807 and never reaches OutOfFlowInsetsForGetComputedStyle().

Minimal page content that triggers the crash

Any page that has an absolutely-positioned element with a non-fixed top/left and calls getComputedStyle on it (directly or via a framework) during a composition tick. The element just needs to exist — it doesn't matter if it was inserted before or after the last paint.
