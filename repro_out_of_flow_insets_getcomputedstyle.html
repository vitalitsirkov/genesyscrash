Required conditions for the crash

1. box->IsOutOfFlowPositioned() == true
      (element has position:absolute, position:fixed, or is OOF in multicol)

2. offset.IsFixed() == false
      (top/left/right/bottom value is: auto, %, calc(), or anchor())
      (a plain px value like "top:10px" returns early at line 807)

3. box->GetLayoutResults().empty() == true
      The LayoutBox was never given a layout result. This happens when:
      a) Document::UpdateStyleAndLayout() returned early before running layout.
         In CEF 6367, UpdateStyleAndLayoutTree() and UpdateStyleAndLayout()
         both return early when UpdateStyleForbiddenScope::IsUpdateStyleForbidden().
         This scope is entered by:
           • LayoutTreeScheduler::OnCompositionStarted()
           • BlinkHelper::GetElementPositionFromViewPortById()
           • BlinkHelper::GetOverlayingElementsById()
           • BlinkHelper::GetAbsoluteDeltasFromFgid()
           • VRMHelper::GetElementLeftAndTop()
           • VRMHelper::GetElementWidthAndHeight()
         Any of these holding the scope while getComputedStyle().top is called
         (directly or indirectly by page JS) will trigger the crash.
      b) The element is newly created (display:none→block or fresh createElement)
         and no layout cycle has run since.

4. style.MayHavePositionFallbackList() must be false for the fallback path
   at line 841–844 to be skipped (i.e. the element does NOT use position-try-options).

WhatsApp-specific trigger (hypothesis)

WhatsApp Web uses:
  • position:fixed; top:50%; left:50%   (loading spinner / overlays)
  • Dynamic show/hide of overlays via display:none ↔ display:block
  • getComputedStyle() calls for animation calculations
  • requestAnimationFrame callbacks that coincide with CEF composition ticks

The most likely scenario:
  1. WhatsApp inserts/shows a position:fixed overlay (new or re-attached LayoutBox)
  2. A CEF helper (blink_helper / vrm_helper) fires in the same timeframe,
     entering UpdateStyleForbiddenScope
  3. WhatsApp JS calls getComputedStyle(overlay).top
  4. UpdateStyleAndLayout() returns early (scope held) → layout_results_ empty
  5. ValueForPositionOffset() calls OutOfFlowInsetsForGetComputedStyle()
  6. CHECK(!layout_results.empty()) fires → crash

NOTE: To get the exact trigger, a CEF crash stack trace is needed.
      The patterns above cover all known code paths that can reach the CHECK.
